The Problem

South Africa's roads are losing the fight against congestion.

Every major South African city is being held back by traffic infrastructure designed in the 1980s. The cost is enormous — and growing every year.

R90M+

South African metros spend hundreds of millions each year repairing, replacing, and managing traffic lights — with little to show for it. A single vandalised intersection can cost R300,000 to restore.

Every traffic light in the current system operates entirely in isolation. No light knows what the next one is doing. There is no coordination, no prediction, no adaptation — just a fixed timer that has not changed in years.

The Solution

A living network — four layers working as one.

Sentinel is not a single product. It is a complete intelligent traffic ecosystem, built in four layers that work together from the physical hardware at each intersection all the way to the driver in the car.

Layer 01

The Smart Intersection

At the heart of Sentinel is the Honeycomb Cell — a compact intelligent module that replaces or upgrades the control unit inside each existing traffic light cabinet. Rather than following a fixed timer, the Honeycomb Cell reads live traffic conditions through sensors, makes decisions using on-device artificial intelligence, and communicates directly with neighbouring intersections. Each one operates independently, so a failure anywhere does not bring the system down. If power is lost, it falls back to a safe fixed cycle automatically. Think of it as giving every traffic light a brain of its own.

  • Installed inside the existing signal cabinet — no road works, no new poles
  • Reads vehicle queue lengths, traffic density, and arrival rates in real time
  • Makes timing decisions locally, without waiting for instructions from a server
  • Falls back to a safe pre-programmed cycle if any component fails
Layer 02

The Mesh Network

Honeycomb Cells do not operate in isolation. Each one broadcasts its current state and traffic conditions to its immediate neighbours — the intersections to the north, south, east, and west — every few seconds. When one intersection sees a queue building, it warns the next one downstream so it can prepare. When an emergency vehicle enters the network, a wave of green lights forms ahead of it automatically, block by block, without any central command being needed. This neighbour-to-neighbour communication is what creates the city-wide intelligence — no single point of failure, no central dependency, just a web of intersections that reason together.

  • Intersections share live queue data, signal states, and predictions with each other
  • Green wave corridors form automatically along busy routes during peak hours
  • Emergency vehicles receive priority through signed corridor tokens passed ahead of them
  • The network continues operating even if the cloud connection is unavailable
Layer 03

The Cloud Intelligence

While the mesh handles real-time decisions, the Sentinel Cloud handles long-term learning. All intersections send anonymised traffic summaries to the cloud platform, which uses machine learning to identify patterns — morning rush on a Tuesday, December holiday volumes, the traffic spike after a major event at Moses Mabhida. These patterns are fed back to intersections as updated timing policies, so the network gradually becomes smarter over months and years. The cloud also powers the operations dashboard that city engineers and maintenance teams use to monitor the health of every intersection in the city.

  • Learns seasonal patterns, peak-hour behaviour, and event-driven spikes
  • Sends improved timing policies back to intersections as it learns
  • Powers a real-time city operations dashboard for municipal teams
  • Stores no personal data — only anonymised, aggregated traffic counts
Layer 04

The Driver App

Sentinel's intelligence extends to the people driving through it. The Sentinel app gives drivers something no navigation app currently can — traffic data sourced directly from the infrastructure itself, not estimated from GPS traces. Drivers can see which intersections are operational, which are in fault mode, where congestion is building, and receive alerts when an emergency vehicle is nearby. A premium tier offers green-wave routing, telling drivers the precise speed to maintain to hit a sequence of green lights without stopping. This is the consumer-facing product that turns Sentinel from a municipal infrastructure play into a daily-use brand.

  • Live status of every intersection — operational, degraded, or offline
  • Real-time congestion heatmap drawn from infrastructure data, not GPS estimation
  • Emergency vehicle alerts so drivers know to create a clear path
  • Premium green-wave routing: the exact speed to sail through without stopping
Core Technology

The Honeycomb Cell — three channels, one module.

The Honeycomb Cell is Sentinel's patent-pending hardware module. What makes it unique is its tri-channel architecture — three completely independent communication systems running simultaneously inside a single unit the size of a small hardcover book.

Channel 01

Peer-to-peer mesh

The reflex channel. Runs the DDS real-time protocol used in autonomous vehicles and industrial robotics. Intersections share live queue data with neighbours every one to five seconds, negotiate timing adjustments through a weighted consensus algorithm, and coordinate green waves — all without any involvement from a central server. Sub-50 millisecond latency. If the internet goes down, this channel keeps the city coordinated.

Protocol: Cyclone DDS over WiFi mesh / LoRa — Latency: <50ms
Channel 02

Relay and duplication

The connective channel. Each Honeycomb Cell does not just receive peer messages — it duplicates and relays them further into the network. This is what allows a signal from one side of the city to propagate across the mesh without requiring a central router. Built on the Zenoh data fabric protocol, this channel also bridges the local mesh to the cloud, translating between the real-time peer protocol and the cloud communication format seamlessly.

Protocol: Zenoh data fabric — Role: DDS-to-MQTT bridge and relay
Channel 03

Cloud uplink

The memory channel. Sends periodic health summaries, traffic logs, and AI decision traces to the Sentinel Cloud over LTE, 5G, or fibre — whichever is available. Receives updated timing policies and model improvements in return. This channel is deliberately kept separate from the real-time mesh, so a cloud outage or slow network connection has zero impact on the intersection's ability to function and coordinate with its neighbours.

Protocol: MQTT over LTE-M / 5G / fibre — Encrypted: AES-256 + mTLS
